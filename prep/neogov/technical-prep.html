NEOGOV Technical Interview Prep — Comprehensive Question Bank

Built from a 5-specialist committee + synthesis · For Thom Ives, Revenue AI & Automation Architect

This doc is a reference for understanding, not a script to memorize. Each question has a layered answer:

  • Lingo version — what the technical interviewer expects to hear
  • Conceptual version — plain-English translation with analogies
  • Background — why the interviewer asks this question and what they're testing
  • Experience anchor — which of your specific work proves you've actually done it
  • Red flags — common weak answers to avoid

The committee's strategic synthesis notes are at the top — read those first; they tell you which questions matter most.

How to use this doc

Two passes:

Pass 1 (one sitting, ~90 minutes): Read the strategic synthesis (next section), then skim every question's background + conceptual answer to build the mental map. Don't try to memorize.

Pass 2 (over several days): Drill the priority questions (Section: Weekend Prep Priority Order). Practice the load-bearing answers out loud until they sound natural.

You learn things when you need them. This doc gives you the map so when an interviewer asks, you can navigate confidently — not because you memorized the answer, but because you understand the territory.

  • Don't lead with model preferences (GPT vs Claude vs Gemini) — junior tell

Category 1: System Architecture & End-to-End Design

Why this category matters: This IS the job. The whiteboard end-to-end test will come up in the technical loop; failing to sketch a coherent five-layer architecture in 15 minutes is a disqualifier. Every other technical question downstream depends on this foundation.

Q1.1 — Whiteboard the end-to-end architecture

Lingo version of the question:

"Whiteboard the end-to-end architecture for the Revenue AI Operating System. Salesforce, Outreach, Gong, Tableau, 6Sense, Pursuit, Starbridge feed in; CRO brief, Director scorecard, seller action sheet, and Marketing Intelligence brief come out. Show me your layers, where the LLM substrate sits, and where the cut lines are between ingestion, modeling, intelligence, and delivery."

Conceptual version (if asked in plainer terms):

"Sketch the whole system on the board — data comes in from these seven sources, four reports go out to four different audiences. Show me how you'd separate the layers."

Background — why they ask: This is the literal job description. The hiring manager wants to see whether you can hold the whole system in one head. The technical panel wants to see whether you'll draw clean layer separations or muddle everything into one big "AI does the magic" box. A candidate who can't sketch the data flow in under 15 minutes can't own this role.

Strong answer shape (lingo):

Five clean layers, drawn left to right:

  1. Source connectors / CDC from Salesforce, Outreach, Gong (transcripts + metadata), Tableau-feeding-warehouse, 6Sense / Pursuit / Starbridge
  2. Bronze landing zone (lakehouse) — raw schemas preserved with lineage
  3. Silver conformed layer — revenue entity model: Account, Opportunity, Contact, Engagement, Signal — reconciled across all systems
  4. Gold marts — one purpose-built mart per artifact (CRO mart aggregates company-wide; Director mart filters by org node; seller mart filters by assigned accounts; Marketing mart aggregates by segment/persona)
  5. AI intelligence layer — reads from gold, retrieves from Gong vector index for unstructured context, runs structured-output LLM passes, writes typed artifacts to a delivery layer (Salesforce embedded, Outreach task, Slack, email, PDF)

Call out explicitly:

  • Orchestration plane (Dagster / Airflow / Fabric pipelines) running the DAG
  • Eval harness sitting beside the LLM layer, not as an afterthought
  • Human-in-the-loop checkpoints for the high-stakes CRO brief
  • Monthly signal pack as its own batch pipeline distinct from per-rep daily/weekly artifacts (different SLOs, different freshness requirements)

Conceptual version of the answer (plain English with analogies):

Think of it like a five-station assembly line. Station 1 is the loading dock — data trucks arrive from seven different vendors with seven different package shapes; you accept whatever they ship and timestamp the arrival. Station 2 is the receiving room — you keep the raw packages in case you need to audit, but you don't try to use them yet. Station 3 is the standardization bay — every package gets opened, contents normalized into the same vocabulary so you can talk about "this account, this opportunity, this person" without arguing about which system's name is right. Station 4 is the staging warehouse — same data, but pre-arranged into four different shelves: one for each audience (executive, manager, seller, marketing). Station 5 is the briefing room where the four reports get composed and shipped to the people who need them.

The AI lives in Station 5. The lakehouse is Stations 2-4. The orchestrator is the foreman that keeps the assembly line moving. The evaluation system is the QA station that catches bad output before it ships to the customer.

Experience anchor: Your UL Prospector TOGAF work for unstructured materials data at scale is the closest direct analog. Your current Microsoft Fabric work uses this exact layered pattern — Delta lakehouse + PySpark notebooks + AI Functions classifier + structured outputs + cost/latency management. You can describe this from memory because you're doing it daily.

Red flags to avoid:

  • Drawing a single box labeled "AI" between sources and outputs
  • Skipping the conformed entity layer and going straight from raw to LLM ("we'll just RAG over everything")
  • No orchestration plane
  • No eval scaffolding
  • Treating all four artifacts as one pipeline
  • Confusing the AI workspace (runtime) with the data warehouse (substrate)

Q1.2 — Monolithic agent vs. independent agents vs. shared substrate

Lingo version:

"The four artifacts share a lot of upstream work — pulling Salesforce, summarizing Gong, joining 6Sense intent. Do you build one monolithic agent that produces all four, four independent agents, or a shared-substrate plus per-artifact composers? Defend the choice on reliability, cost, and debuggability — not on what's fashionable."

Conceptual version:

"Four reports, lots of overlapping prep work. Do you build one big AI that does all four, four separate AIs that each do one, or something in between? Why?"

Background — why they ask: This tests system-design discipline at the workflow level. The wrong answer is "one big agent that does everything" OR "four totally independent agents." The right answer recognizes shared upstream work, separate composition, and the operational realities of debugging when one artifact is wrong but the other three are fine.

Strong answer shape (lingo):

Shared-substrate architecture. A deterministic pipeline does the heavy structured work once: pull Salesforce snapshot, embed and classify Gong calls for the period, join 6Sense / Outreach data, produce a canonical "opportunity-period" fact table. Then four per-artifact composers each take their slice, apply audience-specific prompts, and produce structured output.

Reasoning:

  • Shared substrate is testable and cacheable — one expensive Gong-extraction feeds all four
  • Per-artifact composers are independently versionable and debuggable — a regression in the Director scorecard prompt doesn't put the CRO brief at risk
  • Idempotency, retry semantics on partial failure, observability per stage

Discuss orchestration tooling (LangGraph, Prefect, Temporal, Dagster) with tradeoffs — but resist tool worship.

Conceptual version of the answer:

The right answer is the same answer a kitchen would give. You don't have one chef doing every dish from scratch (slow, fragile, hard to debug when one dish is wrong). You don't have four chefs each peeling their own potatoes (wasteful, inconsistent — what if one chef's potato is bigger than the others?). You have a prep station that does the common work — peel, chop, blanch — and then four chefs who each compose their own dish from the shared mise en place. If one dish comes out wrong, you debug just that station; the prep station and the other three dishes are unaffected.

Experience anchor: Your multi-agent orchestration work at AI-Strategy + the Echo Outlook AI Add-Ins where you handled both sales-rep and carrier sides from shared marketplace data are the proof points. You designed shared-data, audience-specific-output patterns at production scale before.

Red flags to avoid:

  • "One agent with tools" — monolithic agents are debugging nightmares at this scope
  • Four totally independent agents that each re-pull Salesforce — wasteful and inconsistent
  • No discussion of observability or per-stage debugging
  • Treating framework choice (LangGraph vs CrewAI) as the design rather than the design as the design

Q1.3 — Lakehouse vs. warehouse vs. lake

Lingo version:

"Lakehouse vs warehouse vs lake for this system — make the call and defend it. Structured CRM/engagement data, Gong transcripts and audio, 6Sense intent at account-day granularity, Tableau extracts — where does each land, what's the storage format, and what's the query path for the AI workspace?"

Conceptual version:

"You've got structured data, transcripts, intent feeds, and reports. Where do you put each of them and why?"

Background — why they ask: This reveals whether you think in 2019 patterns (warehouse-only, dump everything into Snowflake) or 2026 patterns (lakehouse with open table formats, vector co-location, semantic layer). NEOGOV likely has Snowflake or BigQuery; the question is whether you can extend it for AI workloads without forking a second stack.

Strong answer shape (lingo):

Lakehouse, with explicit reasoning:

  • Structured CRM / engagement data → Bronze → Silver → Gold in Delta/Iceberg open table format
  • Gong transcripts → object storage with parallel chunked + embedded vector index (call_chunks table with foreign keys back to opportunity_id and account_id so RAG can join structured features to unstructured context)
  • 6Sense intent → high-cardinality time-series; partition by account_date with retention tiered
  • Query path for the AI workspace: structured retrieval via SQL against gold marts → vector retrieval for transcript context → LLM call with both as grounding
  • Semantic layer (dbt metrics, Cube, Fabric semantic model) so the same definition of "qualified pipeline" is used by Tableau, the LLM, and exec briefs — one source of truth per concept
  • Cheap object storage for transcripts; hot warehouse compute only for gold marts

Conceptual version of the answer:

Think of it like a research library. Structured data — the indexed catalogs, the cross-references — lives in the regular library where you can run quick queries. Transcripts and recordings — the long-form documents — live in archives where you can pull them when needed but don't pay to keep them on the active shelves. The semantic layer is the librarian who knows that "qualified pipeline" means the same thing whether you're asking from the AI room or the dashboard room — they don't give you different answers depending on who's asking.

The reason you want both — the catalog AND the archives — is that the AI sometimes needs the indexed structured facts ("this opportunity is at stage 3") and sometimes needs the unstructured context ("the customer said in the last call that procurement is the bottleneck"). A vector-database-only answer is like a library where everything is filed by vibes — fine until you need to know the exact title.

Experience anchor: Your Microsoft Fabric work with Delta lakehouse + PySpark notebooks is exactly this pattern. Your UL Prospector TOGAF architecture for unstructured materials data was a similar mixed-modality problem solved at scale. You can speak with conviction.

Red flags to avoid:

  • "Snowflake for everything" without acknowledging unstructured/vector workloads
  • "Just put it all in a vector DB" without a structured entity model
  • Separate ML feature store + separate semantic layer with no integration
  • No partitioning, retention, or storage tiers
  • Doesn't know what an open table format is or why it matters

Q1.4 — Strategic substrate decision

Lingo version:

"You walk in day one. Do you build this on top of an agent framework (LangGraph, AutoGen, Pydantic AI, custom), or do you bet on a substrate like Snowflake Cortex, Databricks Agent Bricks, or Salesforce Agentforce given the EQT thesis and the existing stack? What's the decision tree, and where would you push back on whichever direction NEOGOV is currently leaning?"

Conceptual version:

"Build on raw AI tooling and own everything, or buy into a platform vendor's AI substrate? What's your decision logic?"

Background — why they ask: Senior-architect strategic question. NEOGOV was just acquired with an explicit AI thesis; pressure exists to move fast and demonstrate AI capability to PE owners. The interviewer wants someone who can hold strategic and technical considerations together — not someone with a favorite framework, and not someone who rubber-stamps the platform vendor. Getting locked into the wrong substrate is a year-long mistake.

Strong answer shape (lingo):

Decision tree explicit:

  1. Time-to-first-value — PE thesis suggests months, not quarters
  2. Where does the data live? Salesforce-heavy → Agentforce has gravity but locks you in; Snowflake → Cortex gets close to data but limits model choice; Databricks → Agent Bricks improving but young
  3. Multi-model optionality matters — the landscape will shift twice in the next year; substrates that lock you to one model provider are real risk
  4. Custom on LangGraph / Pydantic AI gives max control + best observability but costs engineering-team-months
  5. NEOGOV team size (~200-500 employees) — framework leverage matters; can't run heavy custom infra without a real platform team

Land on hybrid pragmatic: Ingestion + signal aggregation on the warehouse close to the data; agent orchestration as a thin custom layer over a mature framework for multi-model flexibility; MCP servers exposing tools across both substrates so neither layer is locked in.

Push back on Agentforce-everywhere or Cortex-everywhere as platform-vendor-coupling risks.

Conceptual version of the answer:

The question is really: do you rent a kitchen or build one? Renting (Agentforce / Cortex / Agent Bricks) lets you cook tonight but the landlord controls what ingredients you can use and raises rent when they want. Building your own kitchen costs more up front but you can switch suppliers when one gets expensive or releases something that hurts your recipes.

The honest answer for NEOGOV: the PE owners want results in months, so renting some of the kitchen makes sense — but the highest-value, hardest-to-replace parts (your signal taxonomy, your evaluation harness, your prompt architecture) you build yourself so you're not locked in when the next better model arrives. MCP is the protocol that lets you have it both ways — tools you build are usable inside whatever rented kitchen you happen to be cooking in this quarter.

Experience anchor: Your PostalDataPI was a concept-to-production single-handed build with MCP-native architecture — you've made these substrate-level decisions under real production constraints. Reference the MCP design choices specifically — they're the proof you understand framework portability.

Red flags to avoid:

  • "I'd use [favorite framework] because it's best"
  • No platform-lock risk discussion
  • Doesn't account for PE-acceleration pressure
  • Doesn't ask about existing team's capability to maintain custom infra
  • Hand-waves multi-model optionality
  • Won't push back on a wrong direction because they want the job

Q1.5 — Role-specific projections from one substrate

Lingo version:

"The four artifacts are role-specific by design — CRO needs aggregate strategic signal, Director needs team-level diagnostic, seller needs account-specific next action, Marketing needs share-of-voice and segment trends. Show me the architecture that produces all four from one substrate without four separate pipelines and without one bloated mega-prompt."

Conceptual version:

"Four very different reports, four very different audiences. How do you produce all four from the same foundation without either duplicating work or building one giant tangled thing?"

Background — why they ask: Tests whether you think in clean abstractions or hacks. The naive approach is four pipelines (expensive, drift) or one prompt with branching (unmaintainable). The senior approach is shared substrate + role-specific projections + per-artifact synthesis prompts. Also reveals product-thinking — do you understand what each persona actually needs?

Strong answer shape (lingo):

  • Shared substrate: signal table + entity model + gold marts computed once
  • Role-specific projection layer: materialized views per persona scope
    • CRO mart aggregates company-wide
    • Director mart filters to org node
    • Seller mart filters to assigned accounts
    • Marketing mart aggregates by segment/persona/competitor
  • Each artifact gets its own synthesis prompt + structured output schema tailored to its consumption mode
    • Executive bullets for CRO
    • Diagnostic tables for Director
    • Ranked action list for seller
    • Trend narratives + Share-of-Voice charts for Marketing
  • Shared retrieval helpers (top-N signals by recency, top-N at-risk opportunities) reused across artifacts
  • Templating layer separates LLM-generated reasoning from deterministic data — numbers, dates, names come from SQL, NOT the model; the model only writes connective tissue
  • Numbers must be auditable; narrative is persona-appropriate
  • Acknowledge: seller action sheet is highest-stakes because behavior change is measured there, so it gets the tightest eval discipline

Conceptual version of the answer:

Think of it like a newsroom that publishes four editions of the same paper for four different readerships. There's one news desk that gathers the facts (the substrate). There are four editors who each decide what their readers need to see — the executive edition gets the strategic summary, the manager edition gets the team-by-team breakdown, the sales edition gets next-step recommendations, the marketing edition gets the trend pieces. They share the news desk's reporting; they don't share editorial voice.

The critical piece: the numbers in every edition come from the news desk's verified facts — never made up by the editor's writing assistant. The editor only chooses which facts to highlight and writes the narrative around them. That separation is what keeps the executive edition trustworthy.

Experience anchor: Your Microsoft Fabric work uses this exact pattern — structured outputs separating deterministic data from synthesized narrative. The Echo Outlook AI Add-Ins had two audience-specific outputs (sales reps + carriers) from one marketplace substrate.

Red flags to avoid:

  • Four separate pipelines with no shared substrate
  • One giant prompt with role switches
  • Lets the LLM generate numbers (hallucination risk on metrics is unacceptable for a CRO brief)
  • No projection layer
  • Doesn't distinguish deterministic data from synthesis text in the artifact
  • Treats all four artifacts as equally important — no prioritization for the behavior-change-measured one

Category 2: Salesforce & RevOps Data Foundations

Why this category matters: The system lives on top of Salesforce. A senior architect who can't name core SFDC objects, doesn't know how OpportunityHistory works, or can't handle entity resolution across SaaS sources can't ship this role. This is where you're most exposed and need the most prep.

Q2.1 — Salesforce data modeling

Lingo version:

"Walk me through how you'd model the data flow from raw Salesforce objects — Opportunity, OpportunityHistory, OpportunityLineItem, Account, Contact, OpportunityContactRole, Task, Event — into a CRO operating pack. Where do you do the heavy lifting: Salesforce reports, a warehouse layer, or your AI workspace? And how do you handle the fact that OpportunityHistory only snapshots on stage/amount changes, not on every field edit?"

Conceptual version:

"Walk me through how data flows from Salesforce up to a CRO report. What objects matter, where does the modeling happen, and what's the trap with OpportunityHistory?"

Background — why they ask: The bedrock test. A senior architect who can't name core SFDC objects, doesn't know OpportunityHistory's snapshot-on-change semantics, or can't articulate the warehouse-vs-CRM tradeoff has no business architecting this role. Nick's technical team will sniff this out in 30 seconds. This is where you lose the job if you wing it.

Strong answer shape (lingo):

  • Name the core objects correctly: Account (the company), Opportunity (the deal), OpportunityHistory (change snapshots — but only on tracked field changes), OpportunityLineItem (products on a deal), Contact (the person), OpportunityContactRole (which contacts are on which opps — the join key for ABM persona analysis), Task (activities to do), Event (meetings)
  • Know OpportunityHistory's snapshot semantics: it only writes a row when specific tracked fields change (stage, amount, close date). Supplement with Field History Tracking, Salesforce Event Monitoring, or CDC for fuller change capture
  • Clear architectural pattern: SFDC as system of record → warehouse (Snowflake / BigQuery) as modeling layer using dbt for slowly-changing dimensions on Opportunity → AI workspace consumes curated marts
  • Tradeoffs: Salesforce reports are fast but break at scale; Tableau is good for dashboards but bad for AI feature stores
  • Nuances: soft-delete / IsDeleted handling, currency conversion in multi-currency orgs, person accounts
  • Bonus: "pipeline-as-of-date" reconstruction (what did the forecast look like on the 15th?) — critical for Director scorecard accuracy

Conceptual version of the answer:

Salesforce is the system of record but it's not a reporting database. The pattern is: keep Salesforce as the source of truth for operational state, but copy it nightly (or via change-streams) into a warehouse where you can model history properly. The trap with OpportunityHistory is that Salesforce only saves a snapshot when certain fields change — not when anything changes. So if a sales rep updates the notes but not the stage, OpportunityHistory has no record. You need to supplement with the Field History tracking or change-data-capture stream to see the full picture.

The AI workspace shouldn't query Salesforce directly — Salesforce has API limits and isn't designed for it. The AI reads from your curated warehouse marts where you've already done the work of cleaning, joining, and modeling history correctly.

Experience anchor: You won't have direct Salesforce experience — that's the honest gap. But you can say: "I haven't operated Salesforce directly, but the structural pattern — system of record + warehouse modeling layer + AI consuming marts — is the architecture I use daily on Microsoft Fabric, and the OpportunityHistory snapshot-on-change semantics are familiar from any event-driven change-tracking system."

Red flags to avoid:

  • Treats Salesforce as a flat table
  • Doesn't know OpportunityHistory is event-driven on specific fields only
  • Runs AI directly against SFDC via SOQL with no intermediate layer (governor limits will kill you)
  • Conflates Account and Opportunity hierarchies
  • Never mentions OpportunityContactRole — THE join key for ABM persona analysis
  • "We'd just use Salesforce Einstein" — that's an Einstein answer, not an architecture answer

Q2.2 — Cross-system entity resolution

Lingo version:

"Salesforce calls an account 'Department of Transportation, State of Texas.' Gong sees 'TxDOT.' 6Sense matches on a different domain. Outreach has three contact records that are the same person. Design the entity resolution layer that makes all four artifacts trustworthy without a human reconciling by hand."

Conceptual version:

"The same account looks different in every tool. The same person appears three times in one tool. How do you reconcile all of this automatically so reports don't double-count or miss things?"

Background — why they ask: Cross-system entity resolution is the silent killer of revenue AI. Without it, signals don't aggregate correctly, briefs cite duplicate accounts as different, and seller action sheets miss obvious context. This is also a govtech-specific challenge: government entity naming is wildly inconsistent ("City of Phoenix" vs "Phoenix, AZ" vs "Phoenix Police Dept").

Strong answer shape (lingo):

  • MDM (Master Data Management) layer with an explicit entity resolution pipeline
  • Deterministic matching first: domain, SFDC ID, normalized name
  • Then probabilistic matching: Splink / Zingg / Senzing-style on (normalized name, address, domain, phone)
  • Canonical entity_id with a crosswalk table mapping every source-system ID
  • Confidence-tiered routing: above-threshold auto-merge, mid-band routed to a human review queue with confidence scores, low-confidence stay unmatched
  • Government-specific normalization: expand acronyms (TxDOT → Texas DOT), handle "City of X" / "X City", recognize County/Parish/Borough denote same tier
  • Versioned — merges undoable with audit trail
  • Downstream LLM pipelines always join through canonical entity_id
  • Bonus: Gong transcripts (unstructured) — fuzzy match call participants to canonical contacts at ingestion, store link with confidence so low-confidence link is visible to LLM as "likely but not certain"

Conceptual version of the answer:

You need a single master list — one canonical "Texas DOT" entity — with translation tables that map "what Salesforce calls it," "what Gong calls it," "what 6Sense calls it" all back to the master. The matching is in tiers. Easy matches (same domain, same Salesforce ID) auto-merge. Probably-the-same matches (very similar name, same address) go to a review queue with a human-readable confidence score. Maybe-the-same matches stay separate until someone confirms. Every merge is logged and reversible — because the worst entity-resolution failure is merging two real customers into one and losing track of that.
