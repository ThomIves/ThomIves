Part 1: What MIDA actually is (from Awais's call)

The 60-second framing

MIDA = Multi-modal Inspection Data Annotation platform, built within Palantir Foundry.

What it does: takes imagery from drones, helicopters, and satellites — and identifies electric assets (poles, transformers, splices, circuits, attachments to the pole) within those images. Then annotates them so human inspectors can review faster.

The point is not to replace the inspector — utilities are regulated and decisions stay human. The point is to do the time-consuming first pass (find the asset, draw the bounding box, suggest the tag) so the human inspector can focus on the judgment calls.

The current model approach (from Awais directly)

"It is a zero-shot model, but we configured the parameters in a way that they are performing like 89 to 90% of the time with accuracy ... We were using Gemini 2.5, 3.1, and it is giving us — we are proudly using Gemini 3.1, and it improved like about 30% from the last month, last update from Google."

What this means in plain terms:

  • Zero-shot model — A model that can identify a new object class without being explicitly trained on it. You give it a description ("a pole-mounted electrical splice") and the model tries to find one in an image. Compare to traditional supervised models that need thousands of labeled examples per class.
  • Gemini 3.1 — Google's current frontier Vision Language Model. Multimodal: it sees images AND produces language responses. Same family as GPT-4V, Claude with vision, etc.
  • Parameter configuration — Even with a zero-shot model, the prompt structure, image preprocessing, confidence thresholds, and post-processing all matter. They're not training the model; they're tuning how it's used.
  • 89-90% accuracy — Strong for zero-shot. The remaining 10-11% is where the team is iterating.

The Golden Pipeline pattern

Foundry engineers built a template-based pipeline — instead of replicating the same code for each asset type (splice, transformer, pole, etc.), they have a single configurable template where an inspector or business person can choose which asset to detect, adjust prompts, and run.

Analogy: Think of it like a customizable photo filter that knows which kind of object you're looking for. You don't write a new filter for every object class; you choose from a dropdown and tune a few parameters.

Two complementary model families

Per Awais's framing, MIDA uses both Vision Language Models (VLMs like Gemini) and traditional trained Computer Vision models (like CNNs — Convolutional Neural Networks).

Why both:

  • VLMs are great for general detection. "Is there a splice in this image? Where?" — VLMs handle the broad identification well because they've seen many things during pre-training.
  • Traditional CV models go deeper. Once you've found the splice, is it corroded? Is there a crack? Is the pole leaning? — these specific condition-assessments need models trained on labeled examples of corrosion, cracks, and lean. VLMs can do this, but accuracy drops; specialized models do better.

The team handling traditional CV models is called Waldo — PG&E's internal team of data scientists working on customized models. Per Awais, "they can go very deep in that. Like from VLM, there is a cut off, you know, part where VLM saying, okay, I can do this, but I cannot go in more details. That's where those trained models come in."

What the work would actually be

If Thom takes the role: backfilling someone from the Waldo team so that person can shift onto MIDA / EPIC work. The shape:

"Bring someone senior who can be a SME [Subject Matter Expert] of the computer vision model, but not just a computer vision model — like a data scientist who can also do some kind of machine learning ops ... help the team put things through from concept to production."

Translation: production CV experience + MLOps fluency is the bullseye combination. The Streak Test Auto-Detector story Thom built for the panel presentation maps cleanly to this — production CV deployed under release-cycle discipline.

Two named use cases driving urgency

Use case 1: Splice and transformer detection for safety risk. PG&E discovered 60,000 splices that weren't known to PG&E through this kind of imagery analysis. Splices near schools, gas stations, or fire-prone vegetation are hazardous. Finding and tracking them was previously a manual inspection problem.

Use case 2: Overhead Inspector / Unauthorized Attachments (AI Accelerator-funded). Communication companies (AT&T, Verizon, cable TV) attach lines to PG&E poles. Authorized attachments require permits. Unauthorized attachments incur CPUC penalties of 500−1,000 each. Estimated 20,000 unauthorized attachments in PG&E's territory — at 1, 000each, that′s * *20M of recoverable revenue.**

MIDA is the platform that surfaces these — both safety hazards and revenue recovery — from imagery.

The Sherlock system context

Inspectors currently use a tool called Sherlock (part of SAP-based platform). Workflow:

  1. Imagery arrives in Sherlock
  2. Inspector reviews, draws bounding boxes
  3. Inspector tags assets with severity (A through F):
    • A = immediate dispatch required
    • B = within 24 hours
    • C, D, E, F = future-scheduled
  4. Inspector adds description ("crack in insulator," "corrosion on insulator," etc.)

The problem MIDA addresses: inspectors often misclassify A and B severity (overcalling), which triggers expensive rapid-response dispatch for what turns out to be a non-issue ("oh, it's just a shadow on the insulator"). Conversely, sometimes they miss real A/B cases. AI-assisted classification helps inspectors review faster AND more consistently.

Decision-making boundary

Critical: Awais explicitly stated "It's still in the human loop, there's no decision making because it's a utility and regulated, so there's no decision making through AI, it's only identification."

This is the same boundary as MIDA's stated principle: "Assistive, not autonomous." When you talk about this work, the framing is always "AI surfaces candidates; human decides."

Part 3: Computer Vision Metrics Deep Dive

This section is the technical depth that Charles Coma will most likely probe. From Thom's Gemini research and standard practice.

The four metrics that matter most

  1. Precision — Of the things the model said were positive, how many actually were?

    • Formula: TP / (TP + FP)
    • High precision = few false alarms

  2. Recall — Of all the things that actually were positive, how many did the model catch?

    • Formula: TP / (TP + FN)
    • High recall = few missed cases

  3. F1 Score — The harmonic mean of precision and recall.

    • F1 = 2 × (Precision × Recall) / (Precision + Recall)
    • The F1 > 0.8 deployment bar that EPIC uses is asking: is the model both precise AND complete enough to actually use?

  4. mAP (mean Average Precision) — The gold-standard object-detection metric.

    • Not just "did you find it," but "did you find it AND did you put the box around it correctly?"

Intersection over Union (IoU)

IoU measures how well the predicted bounding box overlaps with the ground-truth bounding box.

Analogy: Think of two overlapping rectangles. The intersection is the area they share; the union is the total area they cover together. IoU = intersection / union. A perfect prediction has IoU = 1.0 (boxes match exactly). A complete miss has IoU = 0 (no overlap).

Common IoU thresholds:

  • IoU ≥ 0.5 — lenient; "close enough"
  • IoU ≥ 0.75 — strict; "tight match"
  • IoU 0.5:0.95 averaged — used in COCO benchmarks; averages across thresholds

How mAP is calculated

  1. Plot the precision-recall curve for each object class. As confidence threshold drops, recall goes up but precision drops.
  2. Average Precision (AP) = area under that curve for one class.
  3. Mean Average Precision (mAP) = average of AP across all classes.

Two common mAP variants

  • mAP@0.5 (or mAP50) — Calculated at lenient IoU threshold of 0.5. "Did the model find the general location?"
  • mAP@0.5:0.95 — Stricter; averages mAP across IoU thresholds from 0.5 to 0.95 in steps of 0.05. "Was the bounding box precise?"

Why mAP for object detection (not just accuracy)

An image can have dozens of objects of different classes. Simple accuracy ("% of predictions correct") fails because:

  • An image with no splice and no transformer that the model correctly says "no splice, no transformer" has 100% accuracy — but it told you nothing useful.
  • A model that's great on common classes (poles) and terrible on rare ones (splices) can look 95% accurate overall while being useless for what you care about.

mAP measures both classification correctness AND localization precision across all classes, weighted toward what you're actually trying to find.

Practical interview answer for "what metrics would you use?"

"For classification tasks: precision, recall, F1, confusion matrix for per-class behavior. For object detection specifically: mAP at multiple IoU thresholds. For PG&E's safety-critical context, I'd also track per-class recall at fixed precision to make sure we're not trading off rare-class detection for aggregate scores — missing a real fault is much worse than a false alarm in a utility setting."

The validation-set design trap (the Adam-flagged one)

The trap: observation-level random splits when video frames are correlated.

Why it matters: Neighboring video frames are nearly identical. If you randomly split frames into train and test, the model can essentially memorize the train data and look great on test — but it'll fail on new video.

The fix: Split at higher grains — first by time (no leakage across acquisition periods), then by physical asset and camera if possible. This forces honest evaluation.

Synthetic data thread: For rare target classes, synthetic / AI-generated training data can balance the training distribution. Keep the validation set clean of synthetic data. Test set mirrors production conditions — weather, season, time of day.

Vision Language Models (VLMs / VLLMs) — what they are

VLMs = Vision Language Models. VLLMs = Vision Large Language Models (same thing, larger).

These are multimodal foundation models that can both see images and produce language. Examples:

  • Gemini 3.1 (Google) — what MIDA currently uses
  • GPT-4V / GPT-4 with vision (OpenAI)
  • Claude with vision (Anthropic)

How they differ from traditional CV models:

  • Pre-trained on enormous multimodal data — they've seen billions of image-text pairs and learned general visual concepts.
  • Zero-shot / few-shot capable — you don't have to train them on your specific use case to get useful results.
  • Naturally support structured outputs — they can return JSON, categories, descriptions.
  • Trade-offs: cost per query, latency, accuracy ceiling for specialized tasks, lack of fine-tunability.

When you'd use a traditional CNN instead of a VLM:

  • Need very high accuracy on a narrow task (e.g., distinguishing types of insulator corrosion)
  • Need to run at extremely high volume where VLM cost would be prohibitive
  • Need on-edge deployment (drone-based real-time inference)
  • Have a large labeled dataset and the VLM accuracy ceiling isn't high enough

The MIDA-specific picture

Detection / first-pass (VLMs): "Is there a splice in this image? Where? What's the rough condition?" — Gemini 3.1 with prompt engineering.

Deep condition assessment (traditional CV via Waldo): "Is this splice corroded? How severe?" — CNNs trained on labeled examples of corrosion severity.

Production discipline (Thom's lane): evaluation scaffolding, structured outputs, cost / latency management, retraining feedback loops, MLOps. This is where the Microsoft Fabric + ON Semi credentials carry forward.

Q3 — "How do you handle class imbalance in detection?"

Lingo (25 sec):

"Several levers depending on context. Data side: targeted augmentation for the rare class, synthetic generation for hard examples, undersampling the dominant class with care. Loss side: focal loss for detection, class-weighted cross-entropy for classification. Sampling: balanced batches during training. Architecture: for very rare events, consider an anomaly-detection framing rather than a direct classification target. And the evaluation metric must reflect the imbalance — accuracy will mislead you."

Conceptual (20 sec):

"If 99% of your training data is healthy power lines, the model can score 99% accuracy by predicting 'healthy' every time and miss every real fault. The fix is making sure the rare cases get attention — by adding more of them, by weighting their mistakes more heavily during training, or by reframing the problem as anomaly detection where rarity is the whole point."

Experience anchor:

  • ON Semi imaging die defects — defects are rare by design; production ML had to handle the imbalance with discipline
  • HP statistical ML for cartridge wear — degradation events are rare; same general shape

Q4 — "Tell me about VLM choice for auto-labeling — how do you pick?"

Lingo (40 sec):

"Evaluation on a representative held-out set, not on benchmarks. I'd compare GPT-4V, Claude with vision, and Gemini Vision on a few hundred labeled examples that mirror production conditions — comparing accuracy on the actual label schema, structured-output reliability under your output format, latency, cost per call, and rate-limit behavior at expected volume. Then I'd weight those against the practical operational constraints — data residency requirements, vendor risk diversity, contract leverage. The right model often isn't the highest-scoring one; it's the one with the best landed-cost / quality ratio at your operational scale."

Conceptual (25 sec):

"Same principle as picking a tool for a manufacturing line. The shiniest tool isn't always the best. Test the candidate models on real samples that look like what they'll see in production, watch cost and latency at the scale you actually run, and pick the one that's reliable under load. The benchmarks tell you who's good in the lab; your evaluation set tells you who's good for your problem."

Experience anchor:

  • Current Microsoft Fabric AI Functions — chose gpt-4.1-mini against alternatives by running on representative seller-disposition data with full evaluation scaffolding
  • PostalDataPI — single-handed production decisions on stack across 246 countries; this kind of selection-under-constraint thinking is daily

Q5 — "How would you architect a human-in-the-loop labeling pipeline?"

Lingo (50 sec):

"Confidence-tiered routing. The VLM outputs label plus calibrated confidence. Above a tunable threshold, the label goes straight to production with sampled audit. Between two thresholds, the example goes to junior human review. Below the lower threshold, escalation to senior reviewer or domain expert. The reviewer interface shows the model's reasoning along with the label so humans can correct based on rationale, not just guess. Corrections flow back into a continuously updated golden set, which drives both retraining triggers and threshold recalibration. Drift detection on the confidence distribution itself — if the model's confidence pattern shifts, something upstream changed and we need to look."

Conceptual (30 sec):

"The model and the human are partners, not a sequence. The model says 'I'm confident here' or 'I'm not sure here,' and the humans get routed by where their attention is most useful. The high-confidence stuff doesn't need a human; the low-confidence stuff needs the senior eye. The middle is where junior reviewers learn. Every correction the humans make becomes training signal for tomorrow's model."

Experience anchor:

  • Current Microsoft Fabric work uses the same evaluation-loop pattern — structured output + per-row cost / latency management with human-validated golden set
  • ON Semi 3-week release cycle had a similar tiered review structure for manufacturing release decisions

Q6 — "How do you detect drift in a VLM-based system?"

Lingo (35 sec):

"Multiple signals in parallel. First, distribution shifts on the model's confidence outputs — if average confidence drops on a fixed input class, something changed. Second, human override rate — if reviewers start correcting more, drift is operational, not statistical. Third, input-side drift detection on image features — embedding distance between current inputs and the training distribution. Fourth, on the rare-class side, a fixed canary set replayed regularly to catch silent regressions. Alerting tied to thresholds with rate-limited paging so the team isn't drowned by noisy signals."

Conceptual (20 sec):

"Drift is when the world starts looking different than the training data — new weather, new equipment, new season. We watch for three signals: the model getting less confident on familiar things, humans correcting it more, or the input images looking statistically different. Any one of those firing means something upstream changed."

Experience anchor:

  • ON Semi production ML — same drift discipline on a manufacturing line where input conditions shift over equipment cycles
  • HP LaserJet supplies warehouse storage condition modeling — predictive models of environmental variation impact (the same shape as drift detection over time / conditions)
