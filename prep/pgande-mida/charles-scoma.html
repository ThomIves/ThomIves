The most likely 5-7 questions (Charles only has 15 minutes)

Question 1 — "Tell me about yourself"

See above. ~50 seconds. Don't pad.

Question 2 — "Walk me through a CV project you've led end-to-end"

Use the ON Semi Streak Test Auto-Detector story. ~90 seconds:

"At ON Semi I was Test Algorithm Development Engineer on a semiconductor manufacturing line. The biggest RMA-driving defect class was streak defects on imaging die — manual tuning was eating two-plus weeks per product and experienced human inspectors couldn't keep cadence. I built the Streak Test Auto-Detector. First, a parametric illumination model — fifth-degree bivariate polynomial fit to the imaging system's background dome — to isolate signal from the dome's lighting gradient. Then iterative robust regression with outlier suppression to convergence — the same robust-least-squares machinery I'd developed for the M.S. robot calibration work. Then spatial clustering and connected-component analysis on the residual anomaly map, with size-and-density thresholds drawn from related cluster-defect tests. We released it through PITAD's three-week production code release cycle. It removed the two-plus weeks of manual tuning per product, matched experienced human inspectors without fatigue, and stopped being a top-three RMA-driver. The engineering constraint was Python 2.5 plus numpy only — no SciPy — so I wrote the linear algebra and statistical routines from scratch."

The detail signals real depth. The closing constraint detail signals you've actually been in the production trenches.

Question 3 — "How do you think about evaluating a CV model for production?"

Per Adam's intel, this is a standard probe. ~25 seconds:

"It depends on what 'wrong' looks like for the application. For classification, precision and recall and F1, and per-class recall at fixed precision so you're not trading off rare-class detection for aggregate scores. For object detection, mAP at multiple IoU thresholds. For PG&E-scale where missing an actual issue is worse than a false alarm, I'd weight toward recall and watch precision as the budget constraint. And critically — a held-out test set that mirrors production conditions exactly: weather, season, time of day, camera angle."

Question 4 — "How would you set up a validation split for CV with this kind of data?"

~30 seconds:

"The biggest trap is observation-level splits when frames are correlated. I'd split on time first — no leakage across acquisition periods — then on physical asset and camera if possible, so the model isn't memorizing a particular angle or lighting. For sparse target classes, synthetic augmentation and AI-generated training data are options to balance training distribution, but the validation set stays real and mirrors production conditions. A held-out test set you don't touch except for final acceptance."

Question 5 — "VLLMs vs fine-tuned CV models for asset detection — what's your view?"

~40 seconds:

"Depends on label volume, label cost, and accuracy floor. VLLMs win cold-start and sparse-label scenarios — you can label thousands of images with structured-output prompting before training a thing. Fine-tuned models win when you have lots of labeled data, need cost-per-inference at scale, and need low latency. The real answer for a program like MIDA is usually a hybrid: VLLM-based auto-labeling to bootstrap the labeled set, then fine-tune a smaller specialized model for production inference, with the VLLM remaining as the human-review escalation tier. I'd want to understand current label volume and labeling cost before committing."

Question 6 — "How would you approach the MIDA work if you joined?"

CRITICAL: don't propose immediately. Ask questions first.

"Honestly, I wouldn't propose an approach in the first weeks. I'd want to understand: what does success look like at the program level — labels-produced or downstream operational outcomes? What's the data volume and acquisition cadence — real-time, batch, or both? What's the current state — is there a labeling pipeline we're augmenting or is this greenfield? And what's the consequence shape of a wrong answer — what kind of misses are most expensive?"

"Once I had that context, I'd want to look at the foundation-model selection, the structured-output design for auto-labels with confidence and reasoning fields, the evaluation scaffolding with appropriate grain splits, the human-in-the-loop architecture, and the production observability story. But the requirements work comes before the architecture work."

This demonstrates exactly the requirements-gathering posture Charles is testing for. It also signals you respect his program's existing state and won't be a cowboy.

Technical engagement primer — be ready to talk these

Awais explicitly invited technical engagement on CNNs, YOLO, and "whatever you have used." When Charles mentions a specific model or technique in his 2-min brief, light up — don't nod blankly.

YOLO family

  • What it is: Real-time single-shot object detection — input image → bounding boxes + class labels in one forward pass. Anchor-based (YOLOv5/v7) or anchor-free (YOLOv8/v9/v10).
  • Why it matters for PG&E: Industrial CV applications run YOLO heavily for asset detection (poles, transformers, splices). Fast inference, mature ecosystem.
  • YOLO-World (2024): open-vocabulary YOLO — text prompts at inference, no fixed class list. Bridge between traditional YOLO and VLLM-style flexibility.
  • Your honest position: "I haven't deployed YOLO in production myself — my production CV work was the ON Semi Streak Test Auto-Detector, which was iterative robust regression + spatial clustering on the residual anomaly map rather than CNN-based detection. But the deployment-discipline patterns transfer: held-out test sets mirroring production, evaluation by per-class recall at fixed precision, release-cycle change-control, and human-in-the-loop for the long tail. I'd ramp on the specific YOLO variant the team is using quickly because the surrounding production scaffolding is what I bring."

CNNs

  • What you know: Standard convolutional backbones (ResNet, EfficientNet, ConvNeXt). Used as feature extractors for detection (YOLO, Faster R-CNN), segmentation (Mask R-CNN, U-Net), classification.
  • Your honest position: "CNN-literate from PyTorch — used pretrained CNN backbones for study and exploration projects, not production deployment. My production CV at ON Semi was classical — the Streak Test Auto-Detector built on iterative robust regression and spatial clustering on the residual anomaly map. Current production work at Microsoft is LLM-based classification via Fabric AI Functions, not vision. CNN backbones I treat as a well-understood layer in any modern stack; my focus is the production scaffolding around the model regardless of what's under the hood — held-out test sets mirroring production, evaluation discipline, change control, observability."
